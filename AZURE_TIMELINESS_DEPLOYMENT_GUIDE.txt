TFL TIMELINESS AZURE DEPLOYMENT GUIDE (STEP-BY-STEP, NON-DEVELOPER FRIENDLY)
=============================================================================

What this guide helps you do
----------------------------
You will set up:
1) An Azure Storage Account (to host the webpage and data files)
2) An Azure Function App (to collect timeliness data every 2 minutes)
3) An optional admin/debug HTTP endpoint
4) A storage-hosted webpage that reads latest_summary.json directly from storage

This guide assumes:
- You have this repository open in VS Code
- You deploy from VS Code
- You have an Azure subscription


PART 0 - Install tools once
---------------------------
1. Install VS Code extensions:
   - Azure Account
   - Azure Functions

2. Install Azure Functions Core Tools (if not installed):
   - https://learn.microsoft.com/azure/azure-functions/functions-run-local

3. Install Python 3.x if you do not have it.

4. Sign in to Azure from VS Code:
   - Open Command Palette (Ctrl+Shift+P / Cmd+Shift+P)
   - Run: Azure: Sign In


PART 1 - Create Azure resources in the Azure Portal
---------------------------------------------------
You can do this in portal.azure.com.

A) Create Resource Group
1. Search "Resource groups"
2. Click Create
3. Name example: rg-tfl-status-prod
4. Region: choose one region (for example UK South)
5. Create

B) Create Storage Account (for static website + function data)
1. Search "Storage accounts"
2. Click Create
3. Resource group: rg-tfl-status-prod
4. Storage account name: must be globally unique, for example tfltimelinessprod123
5. Region: same region as resource group if possible
6. Performance: Standard
7. Redundancy: LRS is fine to start
8. Create

C) Enable Static Website on this Storage Account
1. Open the storage account
2. In left menu, find "Static website"
3. Enable = On
4. Index document name: index.html
5. Error document path: index.html
6. Save
7. Copy the "Primary endpoint" URL (you will use this as the webpage URL)

D) Create Function App
1. Search "Function App"
2. Click Create
3. Resource group: rg-tfl-status-prod
4. Function App name: example tfl-timeliness-monitor
5. Runtime stack: Python
6. Version: Python 3.12 (or current supported 3.x)
7. Hosting plan: Consumption is usually fine
8. Region: same region as storage
9. Storage account: you can use the same storage account or a dedicated one
10. Create


PART 2 - Get required connection string
---------------------------------------
1. Go to your storage account
2. Left menu -> Access keys
3. Copy Connection string from key1
4. Keep this secure (do not commit it in code)


PART 3 - Configure local settings in VS Code
--------------------------------------------
In your repo folder, create local.settings.json (if missing) with this structure:

{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "<your-storage-connection-string>",
    "FUNCTIONS_WORKER_RUNTIME": "python",

    "STORAGE_CONNECTION_STRING": "<your-storage-connection-string>",
    "TABLE_NAME": "MetropolitanLineDelays",

    "TIMELINESS_STORAGE_CONNECTION_STRING": "<your-storage-connection-string>",
    "TIMELINESS_STORAGE_CONTAINER": "$web",
    "TIMELINESS_INTERVAL_MINUTES": "2",
    "TIMELINESS_WEB_INDEX_TEMPLATE": "web/index.html"
  }
}

Important:
- local.settings.json is local-only and should not be committed.
- For static website hosting, container is $web.


PART 4 - Run and test locally first
-----------------------------------
1. Open terminal in VS Code at repository root
2. Install dependencies:
   pip install -r requirements.txt

3. Run tests:
   python -m unittest tests.test_train_timeliness -v

4. Start function host:
   func start

5. Verify HTTP admin/debug endpoint locally:
   - GET status:
     http://localhost:7071/api/timeliness/admin
   - Force refresh run:
     http://localhost:7071/api/timeliness/admin?refresh=1

6. Confirm local files update:
   - timeliness_data/latest_summary.json
   - timeliness_report.html


PART 5 - Deploy Function App from VS Code
-----------------------------------------
1. In VS Code Command Palette:
   Azure Functions: Deploy to Function App...

2. Select your subscription
3. Select function app: tfl-timeliness-monitor (or your chosen name)
4. Confirm deploy

5. Optional but recommended:
   Azure Functions: Sync Triggers


PART 6 - Add App Settings in Azure Function App
------------------------------------------------
1. Open Function App in Azure Portal
2. Go to Settings -> Environment variables (or Configuration)
3. Add/update these app settings:
   - STORAGE_CONNECTION_STRING = <storage connection string>
   - TABLE_NAME = MetropolitanLineDelays
   - TIMELINESS_STORAGE_CONNECTION_STRING = <storage connection string>
   - TIMELINESS_STORAGE_CONTAINER = $web
   - TIMELINESS_INTERVAL_MINUTES = 2
   - TIMELINESS_WEB_INDEX_TEMPLATE = web/index.html

4. Save and allow app restart


PART 7 - Verify timer + storage webpage
---------------------------------------
1. In Function App -> Functions, confirm TrainTimelinessMonitor exists
2. Wait 2-4 minutes for timer runs
3. Open storage account -> Containers -> $web
4. Confirm files appear/refresh:
   - index.html
   - latest_summary.json
   - timeliness_report.html
   - snapshots/...json

5. Open static website Primary endpoint URL
6. You should see the dashboard page (index.html) loading latest_summary.json
7. Click "Open full timeliness report" to view timeliness_report.html


PART 8 - Use the optional admin/debug endpoint
-----------------------------------------------
Endpoint path:
- /api/timeliness/admin

Actions:
- GET  -> returns status and latest summary (if available)
- POST -> triggers immediate collection + publish
- GET with refresh=1 -> triggers immediate collection + publish

Security:
- Auth level is FUNCTION.
- Use Function Key when calling this endpoint externally.


PART 9 - Recommended cutover from GitHub workflow model
--------------------------------------------------------
1. Confirm Azure timer is updating files in $web consistently
2. Confirm static website endpoint works for your dashboard users
3. Keep monitoring for 1-2 days
4. Once stable, do not use GitHub timeliness workflow for collection/publishing


Troubleshooting quick checks
----------------------------
1. If dashboard says summary missing:
   - Check latest_summary.json exists in $web
   - Check Function logs for publish errors

2. If no timer runs:
   - Verify Function App is running
   - Verify trigger exists and synced

3. If publish fails:
   - Re-check storage connection string app setting
   - Confirm function app identity/networking can access storage

4. If endpoint access denied:
   - Ensure you are using the Function key for FUNCTION auth level


Where this repo now expects things
----------------------------------
- Timer trigger schedule for timeliness: every 2 minutes
- Optional admin/debug endpoint: /api/timeliness/admin
- Webpage source file in repo: web/index.html
- Storage-backed webpage data file: latest_summary.json
